## What is API Design in Requestly?

API Design in Requestly goes beyond just testing endpoints—it's about **creating and documenting API contracts** that serve as a single source of truth for your project. Whether you're planning new APIs, documenting existing ones, or establishing contracts between frontend and backend teams, Requestly helps you structure and organize your API specifications.

<Info>
  **API Design vs. API Testing**: While [sending API requests](/general/api-client/send-api-request) focuses on executing and testing APIs, this guide covers how to **plan, structure, and document** your API architecture.
</Info>

***

## Designing API Request Specifications

Create well-documented API definitions that serve as contracts for your team.

<Steps>
  <Step title="Name Your Endpoint">
    Use descriptive, action-oriented names:
    - `Create User Account`
    - `Get Order Details`
    - `Update Product Inventory`
    
    <img src="/images/design-apis/e45a4c93-9c2d-4b85-a662-ebd3e9ca2079.png" align="center" fullwidth="false" />
  </Step>

  <Step title="Define the Endpoint Structure">
    **HTTP Method**: Choose GET, POST, PUT, PATCH, or DELETE
    
    **URL Pattern**: Use variables for flexibility:
    ```
    {{base_url}}/api/{{api_version}}/users/{{user_id}}
    ```
    
    <img src="/images/design-apis/9a34be36-654e-4124-a9ed-e722785d0700.png" align="center" fullwidth="false" />
    
    <Tip>
      Variables make your API design environment-agnostic and reusable
    </Tip>
  </Step>

  <Step title="Specify Query Parameters">
    Document expected query parameters:
    
    | Parameter | Type | Required | Example |
    |-----------|------|----------|---------|
    | page | integer | No | 1 |
    | limit | integer | No | 20 |
    | sort_by | string | No | created_at |
    
    Add these in the **Query Params** tab
  </Step>

  <Step title="Define Headers">
    Specify required and optional headers:
    ```
    Content-Type: application/json
    Authorization: Bearer {{auth_token}}
    X-API-Version: {{api_version}}
    ```
    
    <img src="/images/design-apis/d415dead-8e81-45c5-8e0c-649a885f4c25.png" align="center" fullwidth="false" />
  </Step>

  <Step title="Document Request Body Schema">
    For POST/PUT/PATCH requests, define the payload structure:
    
    ```json
    {
      "user": {
        "email": "user@example.com",
        "first_name": "John",
        "last_name": "Doe",
        "role": "developer"
      }
    }
    ```
    
    <img src="/images/design-apis/d6f85316-ec64-4478-8ea7-2c8d44971f8c.png" align="center" fullwidth="false" />
  </Step>

  <Step title="Add Business Logic">
    Use [scripts](/general/api-client/scripts) to document expected behavior:
    - **Pre-request**: Data validation, token refresh
    - **Post-response**: Extract values, validate structure
  </Step>

  <Step title="Save Your Design">
    Click **Save** to finalize your API definition
  </Step>
</Steps>

## Using Variables for Flexible Design

Variables make your API designs reusable across different environments.

### Variable Types

<Tabs>
  <Tab title="Environment Variables">
    Switch between dev, staging, production:
    ```
    {{base_url}} = https://api.dev.example.com
    {{base_url}} = https://api.prod.example.com
    ```
    [Learn more →](/general/api-client/environments-and-variables/environment-variables)
  </Tab>
  
  <Tab title="Collection Variables">
    Share values across APIs in a collection:
    ```
    {{api_version}} = v2
    {{service_path}} = /users
    ```
    [Learn more →](/general/api-client/environments-and-variables/collection-and-subcollection-variables)
  </Tab>
  
  <Tab title="Global Variables">
    Use across all collections:
    ```
    {{company_domain}} = example.com
    ```
    [Learn more →](/general/api-client/environments-and-variables/global-variables)
  </Tab>
</Tabs>

### Variable Usage

```
URL: {{base_url}}/{{api_version}}/users/{{user_id}}
Header: Authorization: Bearer {{auth_token}}
Body: {"api_key": "{{api_key}}", "timestamp": "{{timestamp}}"}
```

[Complete Variables Guide →](/general/api-client/environments-and-variables)

## Best Practices

<AccordionGroup>
  <Accordion title="Use Semantic Naming" icon="tag">
    Name APIs based on actions:
    - ✅ `Create New Customer Order`
    - ❌ `POST Request 1`
    
    Good names make your API catalog self-documenting.
  </Accordion>

  <Accordion title="Leverage Variables Everywhere" icon="brackets-curly">
    Use variables for:
    - Base URLs
    - API versions
    - Authentication tokens
    - Resource IDs
    
    This makes designs environment-agnostic.
  </Accordion>

  <Accordion title="Group Logically" icon="folder-tree">
    Organize by feature, service, or version:
    - Feature: Auth, Users, Billing
    - Service: User Service, Order Service
    - Version: API v1, API v2
  </Accordion>

  <Accordion title="Document with Scripts" icon="code">
    Use pre-request scripts to document:
    - Expected input validation
    - Required transformations
    - Token refresh logic
  </Accordion>

  <Accordion title="Include API Versioning" icon="code-branch">
    Plan for change:
    - URL: `/api/v1/users`, `/api/v2/users`
    - Collections: Separate by version
    - Variables: `{{api_version}}`
  </Accordion>

  <Accordion title="Set Collection-Level Auth" icon="lock">
    Configure authorization once at collection level:
    - All child requests inherit settings
    - Override per-request when needed
    
    [Authorization Guide →](/general/api-client/authorization)
  </Accordion>
</AccordionGroup>

## API Design Workflow Example

Here's a typical workflow for designing a new feature's APIs:

<Steps>
  <Step title="Create Feature Collection">
    `+ New` → `Collection` → Name: `User Authentication APIs`
  </Step>

  <Step title="Set Up Environment">
    Create dev environment with `base_url`, `api_version` variables
  </Step>

  <Step title="Design Login Endpoint">
    - Name: `Login User`
    - Method: `POST`
    - URL: `{{base_url}}/{{api_version}}/auth/login`
    - Body: `{"email": "{{test_email}}", "password": "{{test_password}}"}`
  </Step>

  <Step title="Add Post-Response Script">
    ```javascript
    // Extract and save auth token
    const response = rq.response.json();
    rq.environment.set("auth_token", response.token);
    ```
  </Step>

  <Step title="Design Protected Endpoints">
    Create `Get User Profile` with:
    - Header: `Authorization: Bearer {{auth_token}}`
    - Inherits token from login response
  </Step>

  <Step title="Test the Flow">
    Use [Collection Runner](/general/api-client/collection-runner) to test the complete flow
  </Step>

  <Step title="Share with Team">
    Export collection and share or use [Workspaces](/general/team/workspaces)
  </Step>
</Steps>
